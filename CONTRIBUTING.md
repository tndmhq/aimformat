# Contributing

Thanks for your interest in `.aim`. The repo carries the **v0.1 draft** of
the spec and the reference toolkit; breaking changes are possible until
1.0, and design discussion via issues is as welcome as PRs.

Whether you are a human, an AI agent, or a human driving an AI agent, the
path to a good PR is the same:

1. **Read [`AGENTS.md`](AGENTS.md)** — repo purpose, layout, and the working
   conventions (package pinning, planning, memory rules, git etiquette). If
   you use Claude Code, Cursor, or any agent that honors `AGENTS.md` /
   `CLAUDE.md`, it will pick these up automatically.
2. **Skim [`docs/knowledge/`](docs/knowledge/)** (and
   [`docs/log/index.md`](docs/log/index.md) for recent context) so your change
   fits decisions already made.
3. **Open a PR that conforms to the conventions**: focused commits, clear
   messages, pinned dependency versions.

## Development setup

```sh
git clone https://github.com/tndmhq/aimformat && cd aimformat
pip install -e '.[dev]'        # pytest + the interop test deps, all pinned
python3 -m pytest              # the whole suite runs in a few seconds
```

The package is **stdlib-only at runtime** — please keep it that way. Heavy
or optional integrations go behind extras (`[docx]` is the pattern) with
lazy imports and a clear install hint in the raised error.

## How the moving parts fit together

One registry drives everything; most changes touch it plus one artifact:

- **`src/aimformat/registry.json`** — the vocabulary (elements, classes,
  style whitelist, theme slots, event schemas, rule codes). Change it and:
  - regenerate the spec appendix: `python3 scripts/gen_spec_appendix.py`
    (CI fails on a stale appendix);
  - the linter and `aim.css` pick it up automatically (both read the
    registry at import time).
- **`spec.md`** — normative text. Every ` ```aim ` fenced block is a
  complete document that CI lints (`tests/test_spec.py`); intentionally
  invalid examples must use plain ` ```html ` fences. Keep snippets
  canonical — the C001 finding prints the exact expected line when you get
  it wrong.
- **`tests/fixtures/`** — the conformance suite: `ok_*.aim` must lint
  clean, `nok_<CODE>_*.aim` must trigger exactly `<CODE>`. One rule per
  file. Regenerate with `python3 scripts/gen_fixtures.py` (which also
  sanity-checks the expectations).
- **`examples/`** — generated by `python3 scripts/gen_examples.py`; never
  hand-edit the `.aim` files.

## Tests

Every behavior change comes with tests. The suite is organized by concern
(`test_canonical`, `test_document_ops`, `test_proposals`, `test_history`,
`test_lint`, `test_ingest_export`, …); a new lint rule needs its rule code
in the registry, a `nok_` fixture, and a targeted test. Run
`python3 -m pytest` before pushing — CI runs the same thing across Python
3.10–3.13.

## Releasing (maintainers)

The package is live on PyPI: <https://pypi.org/project/aimformat/>. Releases
are wired for [PyPI Trusted
Publishing](https://docs.pypi.org/trusted-publishers/) — no API tokens
anywhere. The one-time setup is **already configured** (since 2026-07-10):
the PyPI project trusts owner `tndmhq`, repository `aimformat`, workflow
`publish.yml`, environment `pypi` (re-add under the project's
**Settings → Publishing** if it's ever lost). Per release:

1. Bump `__version__` in `src/aimformat/__init__.py` (the single source —
   pyproject reads it) and move the `## Unreleased` block in
   `CHANGELOG.md` under the new version heading.
2. `python3 -m pytest` green; `git tag v<version> && git push --tags`.
3. Create a GitHub release for the tag — publishing it triggers
   [`publish.yml`](.github/workflows/publish.yml), which builds sdist +
   wheel and uploads to PyPI via OIDC.

## Notes specific to external PRs

- You are **not required to append to `docs/log/`** — the episodic log is a
  maintainer tool, kept light.
- Changes to **`docs/knowledge/` are maintainer-authoritative**: you may
  propose edits, but expect stricter review there than on code.
- By contributing you agree your contributions are licensed under the
