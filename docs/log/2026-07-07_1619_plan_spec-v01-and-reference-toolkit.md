# Plan: spec v0.1 + reference toolkit

Promote the maintainer's working format definition into this repo as the
normative **`spec.md` v0.1**, deciding the remaining open points (each
judgment call recorded in a companion decision entry for maintainer review),
and ship the **minimal reference toolkit** a developer needs to build on the
format today.

## Deliverables

1. **`spec.md`** — single normative file at the repo root, structured per the
   reference-spec conventions: foreword → introduction → design principles →
   core specification (anatomy, substrate, chunks, proposals, history,
   versioned state vs caches, retrieval, assets, canonical form & hashing,
   distribution) → usage examples → appendix (generated construct reference,
   informative recommendations, future extensions). Every ` ```aim ` snippet
   in the spec is linted in CI (executable spec).
2. **`src/aimformat/registry.json`** — the machine-readable registry (single
   source of truth: elements, class vocabulary, style-prop whitelist, theme
   slots, proposal attributes, event fields, lint rules); generates the spec
   appendix and drives the linter and `aim.css`.
3. **`aimformat` Python package** (src layout, **zero runtime deps**,
   Python ≥ 3.10): mini-DOM + canonical serializer + `doc_hash`; `AimDocument`
   with direct edits (add/modify/delete/move/theme), batches, the pending
   lane (propose/accept/reject/supersede, accept-with-tweaks), undo/redo,
   checkpoints, time travel (`state_at`), `flatten`/`prune`, asset
   pack/gc, summary/TOC/embedding cache helpers, and chain verification;
   registry-driven linter; `aim` CLI (`lint`, `hash`, `new`, `show`,
   `flatten`, `css`); `aim.css` generator.
4. **Tests: ≥ 100** — unit + lifecycle + conformance fixtures
   (`tests/fixtures/ok_*.aim` / `nok_*.aim`, one rule per file) + executable
   spec snippets + generated examples validity.
5. **Docs**: README rewrite (quickstart, API tour), `docs/guide/`,
   CONTRIBUTING additions, CHANGELOG, CI workflow, `.gitattributes`,
   `examples/` generated by the SDK.

## Method

Registry first, then package modules (adapting the maintainer-verified
fixture tooling from the round-1 verification), then spec text, then tests
until green, then docs/examples, then consolidated decision + report entries.

## Addendum (same day): ingestion + DOCX export

Maintainer extended the scope: ship a **DoclingDocument → .aim ingestor** and
an **.aim → DOCX exporter** in v0.1.0, modeled on the editor mechanics
recorded in the maintainer's prior-art notes from a previous project
(chunk-granular ingestion from DOCX/PDF; export that replays pending changes
with a caller-chosen default and can emit real Word revision markup).
Constraints: the ingestor stays dependency-free (it consumes the exported
dict shape; `docling` itself never becomes a dependency), the exporter hides
`python-docx` behind a `[docx]` extra, and both are covered by the test
suite. Pins per the search-then-pin convention: `docling-core==2.86.0`
(dev/test only), `python-docx==1.2.0` (extra).

## Out of scope for v0.1.0 (documented as future work)

MCP server, reference viewer, PPTX/PDF exporters, `aim reconcile`,
`aim open` helper, pagination, cell-level table addressing, slide masters,
`.aimx` container, multi-writer merge, signing, Word-move (`w:moveFrom`/
`w:moveTo`) and theme representation in the DOCX exporter.
